## ✅ 已成功修复

### 修复1: warmup_scheduler的total_iters配置

**问题**: SequentialLR的total_iters使用了epoch数而不是iteration数

**修改前**:
```python
warmup_scheduler = optim.lr_scheduler.LinearLR(optimizer, start_factor=0.001, total_iters=warmup_epochs)
```

**修改后**:
```python
warmup_scheduler = optim.lr.LinearLR(optimizer, start_factor=0.001, total_iters=warmup_epochs * num_batches_per_epoch)
```

**文件**: `train/train_rsprompter_fusion.py` Line 270

**影响**: 确保warmup阶段正确执行，学习率从start_factor×初始LR线性增长到初始LR

---

### 修复2: 提高CosineAnnealingLR的eta_min

**问题**: eta_min = args.lr × 0.001 = 2e-4 × 0.001 = 2e-6，过低

**修改前**:
```python
scheduler = optim.lr_scheduler.CosineAnnealingLR(optimizer, T_max=args.epochs, eta_min=args.lr * 0.001)
```

**修改后**:
```python
scheduler = optim.lr.scheduler.CosineAnnealingLR(optimizer, T_max=args.epochs, eta_min=args.lr * 0.01)
```

**文件**: `train/train_rsprompter_fusion.py` Line 266

**影响**: 最小学习率从2e-6提高到2e-5（10倍提升），避免训练后期学习率过低

## 🔍 问题根因分析

### 为什么sat_backbone学习率过低？

从训练日志分析：

| Epoch | sat_backbone LR | 阶段 | 预期LR | 差异 |
|------|---------------|------|--------|------|
| 1 | 5.00e-06 | Cosine | ~1e-4 | **1/200** |
| 50 | 4.99e-06 | Cosine | ~8e-5 | **1/160** |
| 100 | 2.65e-06 | Cosine | ~1e-5 | **1/38** |
| 101 | 2.69e-06 | Cosine | 2e-6 (接近eta_min) | **符合预期** |

**根本原因**：

1. **eta_min过低**: 2e-6 = 0.000002
2. **CosineAnnealingLR在训练中后期迅速衰减到eta_min**
3. sat_backbone在训练中后期的学习率接近eta_min（2e-6）

**为什么shell脚本的lr_mult未生效？**

经过验证，shell脚本的环境变量和Python argparse的默认值都已经正确设置：
- Shell: `SAT_BACKBONE_LR_MULT_STAGE1=0.5`
- Python: `default=0.5`

**实际学习率计算**：
- sat_backbone: 2e-4 × 0.5 = 1e-4
- sat_other: 2e-4 × 1.0 = 2e-4
- drone: 2e-4 × 0.5 = 1e-4

**但是！从训练日志看，sat_backbone的实际LR是5e-6，不是1e-4！**

可能的原因：

1. **加载了错误的checkpoint**: 可能在stage1训练时学习率被意外降低，stage2加载了该低学习率的checkpoint
2. **SequentialLR的warmup阶段问题**: total_iters配置错误可能导致学习率异常

### RPN训练状态

```
loss_rpn_cls: 0.6850 - 0.6857 (始终接近随机猜测值 0.693)
随机猜测值: -ln(0.5) = 0.6931
```

**结论**: RPN训练严重不足，根本原因是sat_backbone学习率过低
